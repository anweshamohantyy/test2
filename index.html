Project Write-Up

Rationale for Design Decisions

Our central question — is wildfire season starting earlier? — has a temporal dimension (change over years) and a seasonal dimension (which week the season begins). We chose three complementary views to address this from different angles.

Season Start Trend (View 1) uses a dot-and-line plot where each dot is one state-year pair. Lines connect years per state, immediately showing upward vs. downward trends. Color encodes region, grouping states with shared fire ecologies. We considered a bar chart but dots better convey individual data point uncertainty and allow overlapping states to remain legible.

Weekly Fire Heatmap (View 2) shows fire intensity across all 52 weeks for each state in a selected year. This reveals when within a year fires peak, and lets users compare years side by side. We chose a sequential fire-themed color scale (dark → ember) over a diverging one because raw fire counts are inherently unipolar.

Annual Profile (View 3) aggregates all selected states into a single stacked area chart per year. This reveals how the aggregate fire season envelope has broadened and shifted. We considered small multiples but the overlay makes the shift more dramatic and legible.

The threshold slider is a key interactive element — it lets users interrogate our definition of "wildfire season." Rather than asserting a fixed definition, we expose it as a tunable parameter, which is both intellectually honest and pedagogically valuable.

Development Process

The team of four split responsibilities across data, visualization, and design. One member focused on sourcing and pre-processing GOES-16 fire detection records from the NOAA Open Data on AWS (S3 bucket noaa-goes16), aggregating raw NetCDF fire product files into weekly count summaries per state. A second member built the D3 chart components and interaction logic. The third handled layout, CSS, and responsive behavior. The fourth wrote the write-up and coordinated the GitHub Pages deployment.

  • Data Processing: ~10 person-hours (navigating NetCDF schema, AWS CLI, state bounding boxes)
  • D3 Charts: ~14 person-hours (three views, transitions, tooltips, threshold reactivity)
  • Design & Layout: ~6 person-hours
  • Write-up & Deployment: ~3 person-hours
  • Total: ~33 person-hours

The most time-consuming aspect was the data pipeline — the GOES-16 FDCC (Fire/Hot Spot Characterization) product is distributed as one NetCDF file per 5-minute scan interval, requiring us to aggregate thousands of files per year. We ultimately pre-computed weekly summaries and committed them as static JSON, which also keeps page load fast.

Alternatives Considered

We considered a choropleth map as the primary view, but maps struggle to show temporal trends — a user would need to click through years sequentially. Our line/dot view lets all years appear simultaneously. We also prototyped a scrollytelling layout but found it added complexity without aiding the core question.

Data: NOAA GOES-16 Fire/Hot Spot Characterization (FDCC) via AWS Open Data Registry
